      Ha!

      Ok, so, it wasn't that bad. There is a function which 99% of me believes that it was used for the ajax version of the site.

      The function WAS a jquery one, and I am unsure of why it was not working. I merely commented it out and IE seems to be functioning exactly how it was before.

      I'll explain what the function was and why its not being used.

      The Code in Question:
      postdata = $('#postdata').text();
      //$('#postdata').empty(); //Empty it, its sensitive to stupid people.
      //if (postdata) { addLog('harvested postdata!'); addLog(postdata); }

      Basically, This code was written as a dirty bridge between php and javascript. It was designed to have php printout some important postdata and then javascript could find it in the DOM, and then it would remove it.

      How was it Used:
      It was used to allow the survey creation process to talk to the javascript. I believe I used this before when I was using the survey tool inline (via ajax) so, when the forum would post it would include the content generated by the survey creation php. This script would grab the postdata generated.

      NOTE: after a search through vulab.admin.js there is no other references to postdata so I suspect that it was legacy code - especially so as I did not use the ajax inclusion for vulab.

      My Solution:
      1. Well, first I opt'd to use iframes which did not require the same javascript awareness in the survey creation process.

      2. The current system listens for a session variable that is is created when the survey tool is saved in anyway. The session variable (for the survey_id) is thrown so php can discover it, and when any button is clicked a javascript event from the iframe reaches up to the parent document to let it know there was a save and the session variable was created.
