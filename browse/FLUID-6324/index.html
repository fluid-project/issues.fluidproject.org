    Comments

    • Alan Harnum commented 2018-09-17T14:39:30.813-0400

      This problems appears to be because of incompatibility between a global install of the latest docpad and the site code (possibly because one or more of the plugins, as they are pinned at particular versions).

      Using npm install -g docpad@6.79.4 (the last version between the 6.8.x line) resolved this problem for me.

      Needs further investigation.

    • Tony Atkins [RtF] commented 2019-04-17T08:09:34.324-0400

      In my work on FLUID-5722 I confirmed that no version of docpad beyond 6.79.4 works with our plugin stack. The README currently instructs people to run npm install -g docpad, which hasn't worked for more than a year.

      Long term we have discussed migrating away from docpad, but in the short term it should still be possible to contribute to this package without forcing your local install to use an outdated docpad. As a short term solution, I plan to add docpad as a dependency pinned to the last working version and update the README to remove the suggestion to install docpad globally.

    • Tony Atkins [RtF] commented 2019-04-17T08:12:36.297-0400

      In my testing (with an incompatible docpad installed globally), simply adding the dependency works perfectly for things run using npm scripts, since npm adds node_modules/.bin at the start of the PATH.

    • Alan Harnum commented 2019-04-18T17:09:28.384-0400

      Tony Atkins [RtF] - I think we've converged on a similar approach per this comment: https://github.com/fluid-project/infusion-docs/pull/146#issuecomment-484688337, but I think your suggestion would work better since it doesn't rely on an outdated global install of DocPad.

      If you've got existing work do you want to open a PR? Simon Bates has been reviewing some of the work I've been doing to (first) try to upgrade the DocPad version and (second, when it turned out to be a yak-shaving exercise) in finding a way to keep using the older version.
