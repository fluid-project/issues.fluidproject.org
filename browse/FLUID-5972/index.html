    Comments

    • Antranig Basman commented 2016-10-04T10:53:17.750-0400

      The use case is fine, but the implementation strategy is inappropriate - we can't support a fresh re-entry into the top of fluid.model.transformWithRules because there is a whole ecology of other use cases which the transformation system needs to support, including:
      i) The "collectInputPaths" strategy for the entire system
      ii) Support for inverting transforms
      iii) Integration with the modelRelay infrastructure
      etc.

      This should be rewritten to use the standard "expand" or "getValue" primitives that the transformer exposes internally which you can see in transforms which process multiple values such as
      https://github.com/fluid-project/infusion/blob/master/src/framework/core/js/ModelTransformationTransforms.js#L123
      and
      https://github.com/fluid-project/infusion/blob/master/src/framework/core/js/ModelTransformationTransforms.js#L239
      and
      https://github.com/fluid-project/infusion/blob/master/src/framework/core/js/ModelTransformationTransforms.js#L285
